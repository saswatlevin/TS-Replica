We need the following pieces of data:
  * product_id -> Request Body   | From frontend
  * user_id    -> Request Params | From session

updateProductPrice ->
  * Check if the request_body is empty or not, if it is, throw an EmptyRequestBody Error.
  * Check if the product exists or not, if not, throw a ResourceNotFoundError.

  * Get a DEEP CLONE of the request_body
  * Create the filter using { product_id : product_id }.
  * Check if the product_price that you are about to update already exists in the database. -> If yes, throw RedundantUpdateError.
  
  * Use the getProductDiscountCodeAndPercentage() HELPER function to get the existing discount_code and discount_percentage of the given product.  

  * Calculate the discount_amount of the respective Product document as product_price * discount_percentage.
  * Calculate the discounted_total of the respective Product document as product_price - discount_amount.
  
  * Get the updated_product_price from the request_body_deep_clone.
  * Create an update_object = {product_price: updated_product_price, discount_amount: discount_amount, discounted_total: discounted_total}.
  

  * Use the findOneAndUpdate() function with the filter, the update_object, {new: true} and {runValidators: true} and chain it with .lean() to update the product price.
  * Get the result from findOneAndUpdate(). -> RESULT 1

  * Assign the updated_product_price to res.locals.
  * Assign the updated_discount_amount to res.locals.
  * Assign the updated_discount_total to res.locals.

  * Call updateCartItemPrice().
  * Get the result from updateCartItemPrice(). -> RESULT 2
  
  * Call updateCartItemTotals().
  * Get the result from updateCartItemTotals() -> RESULT 3

  * Create an array of RESULT 1, RESULT 2 and RESULT 3.

  * Send the array as res.json.
  * Set the res.status to 200.

NOTE: 
  * res.locals is specifically designed for this purpose — it is an object that persists for the lifetime of a single request-response cycle and is not exposed to the client.
  * res.locals is the preferred choice over req.locals because:
    * It is an officially documented Express property explicitly intended for this use case
    * It is scoped to the current request-response cycle and is automatically garbage collected after the response is sent
    * It is never serialized or sent to the client.
